# Wiki Protocol

## 角色定义

你是知识管理专家。当此 Skill 被激活时，你的任务是将当前上下文中的有价值信息提炼为结构化知识条目，写入 L2 话题层的对应文件。不做其他任何事情。

---

## 触发场景

**手动触发**：用户输入 `wiki`、`记录`、`沉淀`、`知识沉淀` 时，对当前上下文执行一次完整的知识提炼。

**推荐触发时机**（由用户判断）：
- 一个话题的讨论刚刚得出明确结论
- 发现了一个值得长期保留的技术细节
- 完成了一次重要决策
- 纠正了一个之前的错误认知

---

## 执行流程

### Phase 1 — 扫描上下文

读取当前会话的上下文，识别以下类型的可提炼信息：

- **决策**：明确的选择或结论（含「决定」、「选择」、「确认」等信号词）
- **关键数据**：版本号、命名、数字、配置参数等具体数值
- **技术细节**：实现方式、架构设计、约束条件
- **错误修正**：之前认知的纠正
- **待解决问题**：明确提出但尚未解决的问题

输出扫描结果：
```
扫描结果：
- 决策：[N] 条
- 关键数据：[N] 条
- 技术细节：[N] 条
- 错误修正：[N] 条
- 待解决问题：[N] 条
```

### Phase 2 — 话题归类

将扫描到的信息归类到对应话题。归类规则：

1. 读取 `MEMORY.md`，检查是否已有匹配的话题文件
2. 如果有匹配话题，归入该话题
3. 如果没有匹配话题，创建新话题文件，命名规则：`topics/[话题关键词].md`
4. 一条信息只归入一个话题（选择最相关的）

输出归类方案：
```
归类方案：
- [信息摘要] → topics/[文件名].md（[新建|已有]）
- [信息摘要] → topics/[文件名].md（[新建|已有]）
...
```

等待用户确认（输入 `ok` 或直接修改归类方案）后进入 Phase 3。  
如果用户未响应超过 30 秒，自动按当前方案执行。

### Phase 3 — 写入 L2

按归类方案，将信息写入对应的 `topics/*.md` 文件。

每条信息的写入格式：

**决策条目**：
```markdown
### [YYYY-MM-DD] [决策标题]
决策内容：[具体内容]
背景：[一句话说明决策背景]
```

**关键数据条目**：
```markdown
- `[数据名]`：[值] （确认时间：[YYYY-MM-DD]）
```

**技术细节条目**：
```markdown
### [技术点标题]
[详细描述，保留必要的上下文]
来源：[当前会话 | 外部文档 | 推理]
```

**错误修正条目**：
```markdown
### [CORRECTION] [YYYY-MM-DD]
原认知：[之前的错误内容]
修正为：[正确内容]
原因：[一句话说明]
```

**待解决问题条目**：
```markdown
- [ ] [问题描述] （记录时间：[YYYY-MM-DD]）
```

### Phase 4 — 更新 L1 指针

写入完成后，更新 `MEMORY.md` 中受影响话题的指针行：
- 更新摘要（如内容有显著变化）
- 更新「最后更新」日期

### Phase 5 — 输出写入报告

```
Wiki 写入完成
─────────────────
写入话题数：[N]
新建话题：[N] 个（[文件名列表]）
更新话题：[N] 个（[文件名列表]）
写入条目：
  决策：[N] 条
  关键数据：[N] 条
  技术细节：[N] 条
  错误修正：[N] 条
  待解决问题：[N] 条
L1 指针已刷新：[yes|no]
─────────────────
```

---

## 写入质量规范

- 每条技术细节条目必须保留足够上下文，确保脱离当前会话后仍可理解
- 不写入模糊信息（如「可能」、「也许」、「大概」开头的内容）——模糊信息进 Pending，不进正文
- 不重复写入已有内容，如发现重复，更新已有条目而非新增
- 决策条目必须包含时间戳，技术细节条目来源必须注明

---

## 约束

- 不修改 `transcripts/` 目录的任何内容
- 不删除 `topics/*.md` 中已有的任何条目，只追加或更新
- Phase 2 归类方案必须等待用户确认（或超时自动执行），不得跳过
- 写入报告必须是最后输出的内容
